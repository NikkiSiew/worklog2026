## 二、技術架構（全部查證過）

```
你（編輯者）              主管/同事（唯讀）
    │ 登入                    │ 點唯讀連結（不登入）
    ▼                         ▼
Vercel（Next.js App Router）─┬─ 一般頁面（你登入後可編輯）
    │                        └─ /report/[token] 唯讀頁
    ▼                              │
Supabase                          ▼
 ├─ Postgres + RLS          Edge Function（用 service_role 讀）
 ├─ Auth（你的登入）         └─ 驗證 token → 只回傳該分享的頁、唯讀
 └─ Edge Function（PIN 驗證 + 唯讀分享）
    ▲
    │
GitHub Actions（每 3 天 ping，防 7 天閒置暫停）
```

- **前端框架**：Next.js（App Router）+ Tailwind [既有技術棧決策]
- **資料庫**：Supabase（Postgres）[已查證：免費方案含 500MB DB、500MB RAM，supabase.com/pricing 及多來源 2026]
- **身分驗證**：Supabase Auth，免費方案含 50,000 MAU（月活躍使用者）[已查證：supabase.com/docs/guides/platform/manage-your-usage/monthly-active-users 及多來源 2026]
- **部署**：GitHub → Vercel 自動 CI/CD
- **反思頁保護 + 唯讀分享**：Supabase Edge Function（Deno/TypeScript）[已查證：supabase.com/docs/guides/functions]

> **成本事實**：你的情境只有你一人登入，等於 1 MAU，遠在 50,000 免費額度內，Auth 完全免費。
> MAU 只計算「當月實際登入或刷新 token 的不重複使用者」，主管同事走唯讀連結、不登入，不計入 MAU。
> [已查證：supabase.com/docs/guides/platform/manage-your-usage/monthly-active-users]

## 三、帳號模型（v2 新增的地基一）

### 決定：你登入、主管不登入

| 頁面 | 你（登入後） | 主管/同事（唯讀連結） |
|---|---|---|
| 反思頁 | 可讀可寫（且需再過 PIN） | 完全看不到（連結不含此頁） |
| 其他頁（對齊/OKR/記錄/週報/Dashboard/專案） | 可讀可寫 | 只能讀，不能改 |

### 機制（查證後的正確做法）

**你這端**：用 Supabase Auth 登入。登入後你的請求是 `authenticated` 角色，RLS policy 允許你讀寫自己的資料。
[已查證：authenticated 是 PostgREST 在使用者登入後使用的角色，supabase.com/docs/guides/database/postgres/roles]

**主管這端**：不登入、不註冊。透過唯讀分享連結，連結指向一個後端 Edge Function（見第四節）。

### 為什麼不是「讓主管用 anon 角色直接讀」（重要的安全修正）

原規劃書讓其他頁「可分享」，但若做法是對 `anon`（未登入）角色開放讀取，會有結構性風險：

- `anon` 是**全站共用的單一角色**。為了讓主管看「對齊/OKR 頁」而對 anon 開放讀取，這個開放是資料庫層級的，邊界容易擴散到不該讀的表。
  [已查證：anon 是「未登入時 API 使用的角色」，supabase.com/docs/guides/database/postgres/roles]
- 官方最佳實務明確警告：不要只靠 `auth.uid()` 來區分 anon，要用 `TO authenticated` 明確把 anon 排除；混用是常見的 RLS 設定錯誤與資料外洩來源。
  [已查證：supabase.com/docs/guides/troubleshooting/rls-performance-and-best-practices；dev.to 2026 亦點名此為 typical RLS misconfiguration]

**結論**：主管的唯讀存取不走「anon 直連資料庫」，改走「後端函式 + token」（第四節）。這樣反思頁的資料連被誤讀的路徑都不存在。

## 四、分享機制（v2 修正：後端函式 + token）

### 與 v1 的差異

v1 只為「週報」設計了 `/report/[token]` 唯讀連結。v2 把同一個安全模式**擴大到所有要分享的頁**，並明確走後端，而非前端 anon 直連 DB。

### 運作方式

```
主管點連結 /report/[隨機token]
   → Vercel 頁面呼叫 Edge Function，帶上 token
   → Edge Function 用 service_role 在後端讀資料（繞過 RLS）
   → 驗證 token 有效 → 只回傳「該分享的頁」唯讀內容
   → 反思頁資料：此函式根本不查，無外洩路徑
```

- `service_role` 金鑰**只存在伺服器端的 Edge Function**，前端永遠拿不到。
  [已查證：service role key 一律 bypass RLS，且絕不可暴露給前端，supabase.com/docs/guides/troubleshooting/why-is-my-service-role-key-client-getting-rls-errors-or-not-returning-data]
- token 是不可猜的隨機字串。拿到連結的人都能看（業界標準分享連結做法），無密碼。
- 連結只顯示分享內容，無導覽列、無法跳到其他頁、不能改。

> 若日後要加密碼或過期時間，可在 share_links 表加欄位擴充（見第五節）。

## 五、反思頁 PIN 保護（沿用 v1 的「自訂 PIN + 後端真驗證」）

### 為什麼是真保護（事實）
[已查證：supabase.com/docs/guides/database/postgres/row-level-security]

- PIN 的**雜湊值**存在資料庫（不存明碼）。
- 反思資料表開啟 RLS。對 `anon` 角色，啟用 RLS 但不給 policy，請求預設一律被擋。
  [已查證：啟用 RLS 後，未建立 policy 時 anon 請求被預設拒絕，supabase.com/docs/guides/database/postgres/row-level-security；dev.to 2026]
- 前端送 PIN → Edge Function 後端比對雜湊 → 對了才回傳短期 token → 用 token 才能讀反思資料。
- PIN 沒驗證過，反思資料不會離開伺服器，開發者工具也看不到。

### 雙層保護（v2 釐清）

反思頁是「你登入後**再**過一次 PIN」，兩層：
1. 你必須先登入（authenticated）才碰得到應用。
2. 反思頁在登入之上，額外要求 PIN。

### 部署步驟（我會寫詳細說明）
1. 在 Supabase 後台貼上 Edge Function 程式碼（網頁就能做，不必裝 Docker）
   [已查證：CLI 在無 Docker 時自動改用 API 部署，可在 Dashboard 操作，supabase.com/docs/guides/functions/quickstart]
2. 設環境變數存 PIN 雜湊。
3. 完成。

### ⚠️ PIN 忘記了怎麼辦（v2 補充）

PIN 只存雜湊是正確做法，但代表 PIN 忘記就無法還原。重設方式：你到 Supabase 後台改掉那個環境變數（換成新 PIN 的雜湊）即可重設。這是唯一後門，且只有能登入 Supabase 後台的你能做。
[推測：此為「只存雜湊」設計下的標準重設路徑；非外部查證的特定文件，而是該設計的邏輯推論]

## 七、GitHub Actions 自動 Ping（防 7 天暫停）

### 事實依據
- Supabase 免費專案**閒置 7 天後暫停**（compute 停用，**資料不刪**，可從後台喚醒）。
  [已查證：supabase.com/pricing 及 nocode.mba 2026「paused after 7 days of inactivity, no data is deleted」]
- 只要 7 天內有一次真實 DB 活動就重置計時。
  [記憶,未查證：「計時看資料庫活動而非 dashboard 造訪」此細節我未在本輪查到明確官方頁面，標記憶；防呆做法是直接寫 DB，下方 ping 即如此]

### 做法
GitHub Actions 每 3 天對 `keep_alive` 表寫入一次：

```yaml
name: Keep Supabase Alive
on:
  schedule:
    - cron: '0 9 */3 * *'   # 每 3 天 09:00 UTC
  workflow_dispatch:
jobs:
  ping:
    runs-on: ubuntu-latest
    steps:
      - run: |
          curl -X POST "${{ secrets.SUPABASE_URL }}/rest/v1/keep_alive" \
            -H "apikey: ${{ secrets.SUPABASE_KEY }}" \
            -H "Authorization: Bearer ${{ secrets.SUPABASE_KEY }}" \
            -H "Content-Type: application/json" \
            -d '{}'
```

> cron 設 3 天而非 7 天，留安全餘裕。
> [已查證：GitHub Actions 排程在高負載時可能延遲，docs.github.com；此為原規劃書既有標註，本輪沿用]
